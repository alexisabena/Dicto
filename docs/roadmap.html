Everything needed to legally collect money, deduct expenses, and eventually register as a formal business in Mexico. Run this in parallel with the technical foundation — none of it blocks shipping, but it must exist before the first invoice.

Phase 02
Recording + Diarization
Weeks 3 – 5
Upcoming
Deliverable — Doctor records session, gets speaker-labeled Markdown transcript

The core product loop. A doctor opens the web app on their phone, presses Start, records a consultation, and receives a speaker-labeled Markdown transcript within minutes. This is the demo moment for the surgeon friend.

  • Web recorder UI — HTML5 MediaRecorder, chunked upload, VAD auto-pause
  • Start / Pause / Resume button flow + session timer
  • Audio chunk receiver (Node.js) → object storage
  • Python transcription service — Whisper API per chunk, Spanish-first
  • pyannote diarization pipeline — audio → speaker segments → labeled JSON
  • One-time voice enrollment — 30 s sample per speaker (doctor, nurse)
  • Transcript assembler — merge Whisper text + pyannote labels → .md file
Success criterion

Doctor records a 10-minute consultation. Within 5 minutes, a Markdown file appears in the patient folder with speaker labels matching doctor and nurse by name.

Phase 03
WhatsApp Scheduling
Weeks 6 – 7
Upcoming
Deliverable — Patient books appointment end-to-end via WhatsApp

Patients already use WhatsApp for everything. No new app, no onboarding friction. A YAML-defined conversation script handles the full booking flow — readable and editable by the doctor without touching code.

  • Meta WhatsApp Cloud API — business account + webhook verification
  • Node.js webhook handler — receive, parse, route, reply
  • YAML dialogue script loader + state machine
  • Intent detection + slot filling — appointment type, date, patient ID
  • Google Calendar OAuth 2.0 — read availability, write booking
  • Appointment record in Postgres + patient linked
  • Confirmation messages — patient + doctor WhatsApp notifications
Success criterion

Patient sends "quiero agendar" to clinic number. Bot guides them through type, date, and confirmation in under 8 messages. Calendar event appears. Postgres record created.

Architecture · 7 key decisions

Why it's built this way

01

Markdown files as source of truth

Human-readable, Obsidian-compatible, portable. Postgres is the query index — not the primary store for clinical notes. A doctor can always open the folder and read what happened.

02

Self-hosted diarization

pyannote runs on VPS CPU. Slower than GPU, but free and sufficient at MVP scale — roughly 2–3× real-time on 8 GB RAM. No per-minute SaaS fee for speaker separation.

03

Whisper API, not self-hosted

Whisper large-v3 needs ~10 GB GPU VRAM. At $0.006/min it's cheaper to use the API until roughly 200 hours/month of audio — well past the first 50 clinics.

04

Claude Haiku for extraction

Extracting structured fields from a transcript is a well-defined, bounded task. Haiku is 10× cheaper than Sonnet for this workload and fast enough that doctors don't wait.

05

YAML dialogue trees for WhatsApp

The doctor — not a developer — should be able to tweak conversation flows. YAML is readable, editable in any text editor, and version-controlled alongside the codebase.

06

Monorepo on a single VPS

All services in one repo, one server. Easier to deploy, debug, and reason about at MVP scale. Split into separate services when the workload actually demands it — not before.

07

Insurance templates as moat

Collecting and maintaining IMSS, ISSSTE, GNP, AXA, MetLife form templates is the durable competitive advantage. Each format added makes the product harder to clone and more valuable to existing users.
